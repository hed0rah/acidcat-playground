Audio Polyglots & Cavities

Nine file-format-abuse techniques across audio and synth/DAW preset formats. Each is a benign, byte-exact proof of concept in the playground, paired with the acidcat inspect --anomalies rule that detects it. None claims a novel primitive; these are documented or under-documented applications of format structure, and the contribution is the coherent detection side. Maturity is labelled honestly: established, under-documented, or moderate.

01WAV + ZIP polyglot

WAV / ZIPtail-parasiteestablished

Mechanism

A RIFF file's RIFF + 32-bit size header declares the container length; every WAV reader stops there and ignores trailing bytes. ZIP is parsed from the end: tools scan backward from EOF for the end-of-central-directory record. So a ZIP appended after the RIFF boundary yields one file that is simultaneously a valid WAV and a valid archive.

Proof of concept

python polyglot.py wav-zip carrier.wav payload -o poly.wav
unzip -l poly.wav          # the archive is real; the WAV still plays

Detection

acidcat scans the last ~64 KB for a ZIP end-of-central-directory signature sitting past the declared container end, an alert-severity polyglot finding. Prior art: PolyZip, corkami/mitra.

02LSB steganography

PCM WAVin-bandestablished

Mechanism

The least-significant bit of each PCM sample is perceptually inaudible. A payload is written across the sample LSBs, XOR'd with a keystream first so the low bit-plane comes out near-uniform, the profile of an encrypted embed.

Proof of concept

python stego.py embed carrier.wav secret -o st.wav   # any PCM depth
python stego.py extract st.wav -o out                # byte-exact

Detection

acidcat computes per-window Shannon entropy of the low bit-plane. A uniform-high floor is consistent with an embedded payload, but it is statistically identical to a genuine mic/preamp noise floor, dither, or a high-bit-depth capture. So this is reported as a notice describing the entropy, not a claim of stego; separating the cases needs a sample-pair / chi-square test, out of scope here. Decades-old technique.

03OGG dual-bitstream

Oggmulti-streamunder-documented

Mechanism

Ogg (RFC 3533) is a multiplexing container: one physical stream can carry several logical bitstreams, each with its own serial number and beginning/end-of-stream pages. That is how Ogg video ships Theora + Vorbis, but nothing stops two audio codecs sharing a file. A single .ogg with a Vorbis stream and an Opus stream is spec-valid; a single-codec decoder surfaces only its own and the other rides along unsurfaced.

Proof of concept

python ogg_multiplex.py build -o dual.ogg    # a Vorbis tone + an Opus tone
python ogg_multiplex.py analyze dual.ogg     # 2 logical bitstreams

Detection

acidcat counts distinct beginning-of-stream page serials; more than one is flagged as ogg_multistream. This is a spec feature, not an exploit, but under-studied from a hiding-content angle.

04MIDI SysEx / tail carrier

MIDIcavitytailmoderate

Mechanism

A System Exclusive event (F0 ... F7) carries arbitrary 7-bit data addressed to a manufacturer id. Id 0x7D is reserved for non-commercial use, so no synth acts on it, the payload rides inside a structurally valid MIDI event. Alternatively, bytes after the final MTrk chunk are ignored (tail-parasite). This is the same channel hardware synths take firmware updates over. MIDI steganography is well established (velocity/timing: US Patent 7402744; SysEx payload tools like stegano_midi exist); the under-examined part is that no player surfaces a 0x7D SysEx and detection is rare.

Proof of concept

python midi_carrier.py sysex secret -o out.mid   # 0x7D SysEx cavity
python midi_carrier.py tail  secret -o out.mid   # after the last MTrk

Detection

acidcat names the SysEx manufacturer and warns when it is the non-commercial 0x7D id or the payload is oversized; the tail variant is caught by the generic trailing_data rule.

05RF64 / WAV JUNK cavity

WAV / RF64cavityunder-documented

Mechanism

A RIFF JUNK chunk is defined as ignorable padding, and in RF64/BW64 (EBU TECH 3306) it is specifically the placeholder that becomes a ds64 chunk once a file exceeds the 4 GB RIFF limit. Until then its content is free bytes that conformant readers skip. Non-zero content there is a true in-container cavity: the RIFF size field stays honest, and there is no trailing data.

Proof of concept

python junk_cavity.py embed carrier.wav secret -o junk.wav
python junk_cavity.py extract junk.wav -o out    # byte-exact

Detection

acidcat's cavity_content rule flags a JUNK/PAD chunk carrying >= 1 KB of non-zero bytes. The 1 KB floor was calibrated on 2328 real WAVs, where legitimate non-zero JUNK (DAW cue/timestamp metadata) topped out at 641 bytes; the payload PoC was ~3.7 KB.

06Dual-endianness twin

WAV / AIFFcross-endianunder-documented

Mechanism

A 16-bit sample is two bytes; WAV reads them little-endian, AIFF big-endian, so the same on-disk bytes decode to different sample values. Packing one sound into the high byte of each frame and another into the low byte makes the little-endian (WAV) reading one tone and the big-endian (AIFF) reading another; the two files carry byte-identical PCM. There is no single-file dual-container, RIFF (RIFF) and AIFF (FORM) require their magic at offset 0 and collide, so it ships as byte-identical twins. The concept of dual-endianness signals is described in Corkami's format-abuse work; the working, intelligible audio+audio twin is the under-documented artifact.

Proof of concept

python dual_endian.py build -o dual --a-hz 440 --b-hz 880
python dual_endian.py verify dual.wav dual.aiff   # byte-identical PCM

Detection

acidcat computes the lag-1 autocorrelation of the 16-bit samples in both endiannesses. Real audio is structured one way and noise-like byte-swapped; when both readings are strongly structured (threshold 0.9) it is flagged as dual_endianness. On 2328 real WAVs this fired on one (a ~0.3% false rate).

07FLAC APPLICATION / PADDING

FLACcavitymoderate

Mechanism

FLAC's metadata block chain includes an APPLICATION block (arbitrary vendor data keyed by a registered 4-byte id) and a PADDING block (spec'd all-zero). Decoders skip both, so an unregistered APPLICATION id, or non-zero bytes in PADDING, carries a payload while the file decodes normally.

Proof of concept

python flac_cavity.py app carrier.flac secret -o app.flac
python flac_cavity.py pad carrier.flac secret -o pad.flac

Detection

acidcat flags any APPLICATION block (application_block) and non-zero PADDING (cavity_content). The FlacFile utility already uses the APPLICATION block for file embedding, hence moderate, not under-documented.

08MP4 mdat coverage gap

MP4 / M4Acavitymoderate

Mechanism

In ISO-BMFF, audio samples live in the mdat box and are located by the sample tables (stsz sizes, chunk offsets). Appending a payload to the tail of mdat and growing only mdat's size field leaves every existing sample offset valid, no sample references the appended run, so the audio decodes bit-identically. Unlike a free/skip atom or trailing data (which every carver checks), this is a cavity inside a live top-level box. mdat hiding itself is documented (OpenPuff, videostego); what is under-tooled is the coverage-gap detection, summing stsz sizes against the mdat payload, and this offset-free grow-only variant.

Proof of concept

python mp4_cavity.py embed carrier.m4a secret -o cav.m4a
python mp4_cavity.py analyze cav.m4a     # gap == cavity size

Detection

acidcat's mp4_mdat_coverage rule sums every stsz sample size across all tracks and compares to the mdat payload; an unreferenced run (>= 1 KB, and only when samples are actually described, so fragmented/DASH files are exempt) is flagged. It locates moov by a header-only scan, so non-faststart files with moov at EOF are still checked.

09MP3 ID3 padding

MP3cavitymoderate

Mechanism

An ID3v2 tag declares its total size (a syncsafe integer). The region after the last frame, up to that declared size, is padding, spec'd zero. A payload written there is inside the tag's own length (not trailing data) and the audio frames are untouched. The documented ID3 blob features are GEOB/PRIV; raw padding is rarely checked.

Proof of concept

python id3_cavity.py embed carrier.mp3 secret -o cav.mp3
python id3_cavity.py extract cav.mp3 -o out    # byte-exact

Detection

acidcat's id3_padding_nonzero rule walks the frames (skipping the extended header and de-escaping unsynchronisation) and flags non-zero bytes in the padding region.
